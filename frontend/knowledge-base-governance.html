玄学知识库与投资日报子库建设维护方案

面向母知识库、投资日报子知识库、未来多个应用层、多端多人多 Agent 协作的治理方案。最后更新：2026-07-07。

当前定位投资日报子知识库，不是玄学母知识库。
核心原则母库权威，子库应用，同步先 draft，发布需 review。
程序调用默认只读 active + usable_for=daily_report。

1. 总体结论

完整体系应采用三层结构：母知识库、应用层子知识库、运行层知识库。母库沉淀通用玄学理论、SOP、速查表、案例和冲突记录；应用子库只保存具体场景需要的映射、验证、输出和合规边界。

玄学母知识库
  -> 应用层子知识库
      -> 投资日报知识库
      -> 事业/职业知识库
      -> 情感关系知识库
      -> 择日工具知识库
      -> 风水方位知识库
  -> 运行层知识库
      -> 回测结果
      -> 用户反馈
      -> 发布记录
      -> 审计日志

投资日报不能定位为荐股投顾，只能定位为：文化视角 + 市场结构化观察 + 数据确认 + 风险提示。

母知识库目前主要是“秉笔介绍文档”描述的状态：飞书 Wiki/Base 为权威源，本地 references 为兜底镜像，内容覆盖八字、紫微、建除、九星、奇门、月运 SOP、案例和速查表。

3. 职责边界

我在本项目中负责投资日报子知识库：把母库中与 A 股、行业、ETF、市场节奏、风险表达相关的内容转为可调用的应用知识，并维护日报程序、回测解释、AI 对话和合规边界。

我不直接维护整个玄学母知识库，不直接改写理论权威结论，不把未经审核的母库内容直接发布为投资日报 active 知识。

母库回答“玄学体系里这个知识点是什么、怎么用、有哪些来源和争议”；投资日报子库回答“这个知识点在投资日报中能不能用、怎么用、不能怎么用、是否经过验证”。

5. 投资日报产品与商业路径

版本内容边界
免费版每日市场状态、行业风格观察、风险提示、简化版月运/日运解释。
会员版历史回放、行业观察列表、ETF 视角、背离提示、自选行业提醒、月运区间跟踪。
B2B 版社群日报、内容授权、企业内部市场观察卡、定制行业看板、多账号/多群分发。
人工咨询只做内容解释和使用教学，不做个股买卖建议，不代替用户做投资决策。

用户分类

  • 玄学兴趣用户：关注五行、月运、节气叙事。
  • ETF 用户：关注行业 ETF、趋势确认、风险窗口。
  • 个股用户：只能获得行业和风格背景，不输出买卖指令。
  • 稳健用户：关注风险门控、回避方向、仓位提醒。
  • 运营/B2B 用户：关注可复用文案、飞书卡片、社群解释材料。

6. 多人多 Agent 权限分工

角色职责边界
Owner决定 active 知识、商业边界、应用优先级。最终审批。
Mother KB Curator维护母库理论、SOP、来源、冲突、知识缺口。不直接发布应用商业规则。
App KB Curator维护应用子库，投资日报中负责金融映射和输出规则。不改母库权威理论。
Quant/Backtest Agent负责验证和回测。只改验证层和案例层。
Content Agent负责日报表达、推送卡片、用户视角文案。不改 active 算法和理论。
Compliance Reviewer审核免责声明、禁用词、商业边界。对发布有否决权。
Runtime Agent日报程序和 AI 对话运行者。默认只读 active 知识。
Sync Agent负责同步预览、差异检测、draft 生成。不能直接发布 active。

7. 增删改查流程

inbox -> draft -> review -> active -> deprecated
  • 新增：任意人或 Agent 可提交到 inbox 或生成 draft，必须带来源和适用范围。
  • 修改：必须生成 change request，记录修改原因、修改前后内容和影响范围。
  • 删除：不物理删除，只标记 deprecated，保留历史引用。
  • 查询：程序默认只查 active；日报只查 usable_for=daily_report。
  • 审计：每次同步、修改、审核、发布都写入日志。

8. 同步中心方案

同步中心负责不同知识库之间的自动发现、一键导入、审核发布和审计留痕。

飞书 Wiki/Base
  -> 本地镜像
  -> 标准知识 schema
  -> 应用子库 draft
  -> review
  -> active
  -> 程序/API 调用

同步模式

  • Preview：只看差异，不写入。
  • Draft：写入应用子库 draft，不发布。
  • Review：负责人审核。
  • Publish：审核通过后转 active。

投资日报同步规则

app_targets 包含 investment_kb
或 usable_for 包含 investment_daily / daily_report
或 domain 属于 month_fortune / wuxing / jianchu / ziwei 且存在 A 股映射

11. 需要团队确认的信息

  1. 母知识库权威源到底是哪一个飞书 Wiki/Base。
  2. 谁是母知识库 Owner。
  3. 谁有权把知识从 draft 发布为 active。
  4. 投资日报子库是否允许回写母库，如果允许，谁审核。
  5. 当前飞书 Wiki/Base 是否能开放给当前 bridge profile 读取。
  6. 母库里的标签体系是否已有标准，还是由我们新建。
  7. 哪些应用层优先级最高：投资日报、个人命理、事业、情感、择日、风水、内容号。
  8. 投资日报商业化的免费/会员/B2B 边界最终怎么定。
  9. 用户分类是否采用当前五类，是否要加社群运营者或机构用户。
  10. 合规审核负责人是谁。
  11. 是否需要飞书里直接做审核按钮和发布流程。
  12. 是否允许程序定时同步，还是只允许人工一键同步。

12. 推荐下一步

  1. 确认母库 Owner、投资日报子库 Owner、合规审核人。
  2. 建立 inbox、draft、review、active、deprecated 的目录和日志。
  3. 做 sync-knowledge-base.js --mode preview，先只看差异，不写入。
  4. 做知识同步中心页面。
  5. 把日报生成流程改成只读取 active + usable_for=daily_report 的知识，并记录每次日报引用过的知识条目。
